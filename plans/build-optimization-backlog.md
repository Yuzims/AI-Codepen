# 构建体积优化待办清单

> 相关文档：本文用于记录后续构建体积优化 backlog；阶段性性能优化总结见 [task3-summary.md](task3-summary.md)。

## 结论

可以，**完全可以先做其他功能，再回头优化这块**。

当前 `vite build` 的提示是：

> Some chunks are larger than 500 kB after minification.

这属于 **构建告警，不是构建失败**。从当前产物结构看，项目已经完成了首屏相关的重要优化：
- 路由已做懒加载
- 大型运行时依赖已拆分为独立 chunk
- 主入口 chunk 已经比较小

所以这件事更适合作为一项**后续专项优化**，而不是当前必须阻塞功能开发的问题。

---

## 当前现状

目前前端构建产物中，体积偏大的 chunk 主要有：

| chunk | 大小 | 说明 |
|------|------|------|
| `tsWorker` | ~3.4MB | TypeScript worker 内直接打进了 `typescript` |
| `sass-runtime` | ~3.2MB | 浏览器侧运行时编译 Sass |
| `babel-runtime` | ~2.9MB | 浏览器侧运行时编译 React / Babel |
| `codemirror` | ~537KB | 编辑器相关依赖整体仍然偏大 |

说明：
- 这些包**大多已经不是首屏主包**，所以对首页初始加载的影响没有以前那么严重。
- 但它们仍然会影响：
  - 首次进入编辑器页面的加载体验
  - 编辑器相关功能首次可用时间
  - 总构建产物体积
  - 未来继续叠加功能后的维护成本

---

## 为什么现在可以先不做

当前阶段可以延期优化，主要因为：

1. **这是 warning，不是 error**  
   不会阻止构建，也不会阻止功能继续交付。

2. **首屏关键路径已经相对健康**  
   当前主要问题不在首页入口，而在编辑器能力相关的重型依赖。

3. **这类优化更适合集中处理**  
   这不是改一两行配置就能彻底解决的问题，更像一轮“依赖装载策略”优化，适合等功能稳定后统一处理。

4. **如果近期还有功能开发，过早优化可能带来额外回归成本**  
   尤其是编辑器、预览、编译器、worker 这一块，改动后需要重新验证交互与兼容性。

---

## 推荐优化优先级

## P1：优先做（收益最大）

### 1. 将 `tsWorker` 中的 `typescript` 外置或改为运行时加载

**现状**
- `src/workers/tsWorker.ts` 顶部直接 `import * as ts from 'typescript'`
- 这会导致整个 worker chunk 非常大

**问题**
- TypeScript 编译器本体很重
- 即使它被放在 worker 中，也会显著增加首次启用 TS 能力时的加载成本

**建议方向**
- 改为 CDN / 静态资源 / `importScripts` / 运行时加载
- 或者评估是否只保留当前真正用到的 TS 能力

**预期收益**
- 这是当前最值得优先优化的点
- 大概率能显著降低 `tsWorker` chunk 体积

**涉及文件**
- `src/workers/tsWorker.ts`

---

### 2. 统一 Babel / TypeScript / Sass / Less 的加载策略

**现状**
- 有些运行时编译器走打包 chunk
- 有些地方又通过脚本标签/CDN 注入
- 存在“包内一份 + 运行时再加载一份”的潜在重复策略

**问题**
- 维护成本高
- 容易出现重复下载、行为不一致、版本漂移
- 后续排查性能问题时不容易判断真实来源

**建议方向**
- 统一选择一种策略：
  - 要么全部本地 chunk 化并按需加载
  - 要么全部改为稳定的外部静态资源加载
- 同时明确版本来源，避免 Babel / TS 在不同位置使用不同版本

**重点检查位置**
- `src/services/compilerService.ts`
- `src/components/Preview.tsx`
- `src/components/DebugPreview.tsx`
- `src/workers/tsWorker.ts`

**预期收益**
- 减少重复装载风险
- 降低后续维护复杂度
- 有机会继续压缩相关 runtime chunk

---

## P2：第二阶段做（收益明确，但优先级低于 P1）

### 3. 让预览页脱离完整 CodeMirror 依赖

**现状**
- 公开预览页使用了 `ReadOnlyEditor`
- `ReadOnlyEditor` 仍然基于完整 CodeMirror 体系

**问题**
- 如果只是展示代码，只读模式没必要一定使用完整编辑器能力
- 这会把较重的编辑器依赖带到预览场景中

**建议方向**
- 评估是否用更轻量的代码高亮展示替代只读编辑器
- 或者把只读编辑器再单独拆分得更细

**涉及文件**
- `src/pages/PreviewPage.tsx`
- `src/components/ReadOnlyEditor.tsx`

**预期收益**
- 降低公开预览页的加载成本
- 避免“只展示代码也要加载完整编辑器”

---

### 4. 进一步细分 CodeMirror 能力包

**现状**
- 当前 `codemirror` chunk 约 537KB
- 已经独立分包，但仍偏大

**问题**
- 编辑器相关语言支持、补全、lint、view 层能力都比较重
- 可能存在“当前页面未必都需要一次性加载全部能力”的情况

**建议方向**
- 检查是否可以按语言或功能继续拆分：
  - JS / TS
  - HTML
  - CSS / Less
  - Vue
  - autocomplete / lint / readonly
- 识别编辑页与预览页是否真的共享同一套完整依赖

**涉及文件**
- `src/components/Editor.tsx`
- `src/components/ReadOnlyEditor.tsx`
- `src/services/completion/`
- `src/services/lintService.ts`
- `vite.config.ts`

**预期收益**
- 进一步降低编辑器首次加载压力
- 为后续新增语言能力预留空间

---

## P3：可选优化（收益较小或偏工程治理）

### 5. 检查是否仍有“浏览器内编译”可以迁移到服务端或构建阶段

**现状**
- 当前产品形态本身依赖浏览器内编译，这是合理的
- 但可以继续梳理哪些场景是真正必须在浏览器做的

**建议方向**
- 区分：
  - 用户实时编辑预览必须在浏览器编译
  - 非实时分析 / 预处理 / 存储前转换未必必须在浏览器做

**收益**
- 如果能减少运行时编译职责，整体包体和执行成本都可能继续下降

---

### 6. 仅在确认无需进一步压缩后，再考虑调整 `chunkSizeWarningLimit`

**不建议现在做**
- 直接把 warning 阈值调大，只是隐藏提示，不是解决问题

**建议**
- 只有在完成主要优化、并确认这些大 chunk 属于“业务上合理且可接受”之后，再决定是否调高阈值

**涉及文件**
- `vite.config.ts`

---

## 建议执行顺序

后续真正开始做这轮优化时，建议按下面顺序推进：

1. **先处理 `tsWorker` 外置 `typescript`**
2. **再统一 Babel / TS / Sass / Less 的加载策略**
3. **然后处理预览页只读编辑器轻量化**
4. **最后再考虑 CodeMirror 继续细分拆包**
5. **确认结果后，决定是否保留或调高 chunk warning 阈值**

---

## 建议验收方式

优化时建议同时观察以下指标：

1. `dist/assets` 中相关 chunk 的实际大小变化
2. `dist/bundle-stats.html` 的依赖归属是否更清晰
3. 首次进入编辑器页的加载耗时
4. 首次启用 TS / Babel / Sass / Less 功能时的等待时间
5. 是否引入新的兼容性问题或运行时错误

---

## 风险提醒

这轮优化的风险主要在于：
- 编辑器能力依赖复杂，拆分后容易出现功能缺失
- 运行时脚本加载顺序处理不当，可能导致编译器未就绪
- worker / iframe / 动态 import / 外部脚本混用时，调试成本会升高

所以更适合在**功能开发阶段结束后，单独开一个优化任务处理**。

---

## 一句话决策建议

**可以先继续做功能，这块完全可以作为后续专项优化处理。**  
如果后面只选一个点优先做，建议先做：**`tsWorker` 外置 `typescript`**。
